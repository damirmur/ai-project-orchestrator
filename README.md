# AI Project Orchestrator

## Что это
VK‑бот, который передаёт запросы пользователя в локальный LM‑Studio, позволяет модели редактировать файлы в каталоге `projects/` и применять изменения после подтверждения.

## Быстрый старт
```bash
# Установить зависимости
npm ci

# Скопировать пример переменных окружения и заполнить
cp .env.example .env   # отредактировать .env

# Запустить в режиме watch (автоперезапуск при изменении кода)
node --watch --experimental-transform-types src/main.ts
```

## Переменные окружения (обязательно)
| Переменная | Описание |
|------------|----------|
| `VK_TOKEN` | Токен VK‑IO бота |
| `GROUP_ID` | ID группы бота |
| `USER_ID` | ID администратора, которому разрешено писать |
| `CHAT_PEER_ID` | ID чата, в котором бот слушает сообщения |
| `PROJECTS_ROOT` *(необязательно)* | Корень директории, где бот может читать/писать файлы (по умолчанию `./projects`) |
| `LMS_URL` *(необязательно)* | URL локального сервера LM‑Studio (по умолчанию `http://localhost:1234`) |

## Основные команды бота
- `🤖 Модели` – выбрать активную модель LM‑Studio.
- `📂 Файлы` – показать список файлов проекта (по 10 штук).
- `📂 <имя файла>` (кнопка) – выбрать файл, загрузить его содержимое в контекст ИИ.
- `📝 /write` – сохранить сгенерированный ИИ код в выбранный файл (после подтверждения).
- `✅` / `❌` – подтвердить или отклонить запись.
- `🏓 Пинг` – проверить, что бот живой.

## Как работает состояние
* При старте `state.service` читает `state.json` (если файла нет – создаёт пустой).
* Для каждой комбинации `USER_ID:CHAT_PEER_ID` создаётся отдельная сессия, в которой хранятся:
  - `activeFilePath`
  - `stagedContent`
  - `lastAiResponse`
  - `pendingMessage`
  - `lastMessageId`
* Состояние сохраняется в `state.json` после **каждого** изменения, поэтому при перезапуске бот восстанавливает контекст.

## Удалённые файлы/папки
- `.git` и `node_modules` (выполняются `npm ci` при необходимости).
- Файлы `analiz.md`, `architecture.md`, `todo.md`, тестовые файлы в `projects/` (т.к. они не участвуют в работе бота).

## Дальнейшие шаги
- При желании добавить типизацию (`tsconfig.json` → `strict: true`).
- При необходимости написать unit‑тесты для `state.service`.
- При добавлении новых команд — расширять `AGENTS.md`.
